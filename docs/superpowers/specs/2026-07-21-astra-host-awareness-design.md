# ASTRA host awareness (`HOST.md`) — design

**Date:** 2026-07-21  
**Status:** Approved (Approach 1)  
**Repos:** MXPF-ASTRA-AGENT + MXPF-AARIA-API

## Goal

Give each ASTRA minion a durable **host profile** (`HOST.md`) so AARIA knows the machine’s purpose, OS, runtimes, and services — without shipping huge MQTT payloads.

## Decisions

| Topic | Choice |
|-------|--------|
| Purpose | Auto-guess + install prompt override |
| MQTT | Compact `host` summary on announce (~2–4 KiB) |
| Full file | On-demand via allowlisted `host.profile` action |
| Storage on AARIA | `AgentRecord.host` + `data/fleet/hosts/{agentId}.md` + FLEET.md purpose line |

## Minion (`HOST.md`)

- Path: `$ASTRA_ROOT/HOST.md` (never deleted by install/upgrade)
- Generated by `deploy/probe-host.sh` + install-upgrade step
- Sections: Purpose, Identity, Resources, Runtimes & tools, Data services, Notable services, Notes
- Purpose auto-guess (nginx/apache → web; desktop session → workstation; etc.), then confirm prompt
- Optional `--refresh-host` on upgrade

## Announce payload

```json
"host": {
  "purpose": "Web server",
  "os": "Ubuntu 22.04",
  "arch": "x86_64",
  "summary": "<markdown excerpt ≤4KiB>",
  "updatedAt": "ISO-8601",
  "hash": "sha256:…"
}
```

Default caps include `host` when profile exists: `["health","exec","host"]`.

## Cap `host.profile`

- Args: `{ "refresh": false }` — if true, re-run probe then return full `HOST.md`
- Returns: `{ ok, markdown, purpose, updatedAt, hash }`
- Result body capped like other caps (64 KiB)

## AARIA

- `onAnnounce`: persist `host` on registry; write/update `data/fleet/hosts/{agentId}.md`
- `FLEET.md`: include purpose in roster notes / table column when present
- Ops Fleet Detail: show purpose + OS + truncated summary
- Persona: existing `FLEET.md` load picks up purpose; optional short host blurb in fleet skill

## Out of scope

- Continuous auto-rescan on a timer (manual refresh / install upgrade only)
- Shipping secrets, full process lists, or private config files into `HOST.md`
